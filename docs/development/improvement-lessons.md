# Supacrawl Improvement Lessons (experiential loop)

This file is the **memory of the supacrawl improvement process**. It exists so that every "improve supacrawl" session compounds on the last instead of starting cold. An agent asked to improve scrape/search quality reads this file **first**, runs the benchmark as the fitness function, makes targeted changes, re-runs the benchmark to confirm a real lift, then **appends a dated entry** describing what it did, what worked, what surprised it, and what is still open.

This is the developer-facing half of the self-improvement story. The product-facing half — supacrawl learning per-domain strategies in the field — is the `adaptive` feature module (see `docs/product/features/adaptive/`). Keep them distinct: this file makes the _tool_ better over releases; the `adaptive` module makes a _running scrape_ better over repeated hits.

## The loop

1. **Read** this file end to end. Note the open threads and what was already tried.
2. **Measure**: `supacrawl bench run` → record overall score, per-category, and the worst scored cases. `supacrawl bench compare` against the previous run to catch regressions. The bench is the fitness function; do not "improve" without it.
3. **Target**: pick the weakest scored category/case or a real-world gap (drive the CLI + the browser-driver MCP against representative sites; the operator's matrix is Amazon, Qantas, hotels, ATO gov tax-law for RAG, SERP, pet food).
4. **Change**: make the smallest change that should move the metric. Prefer fixing a root cause over a special-case.
5. **Confirm**: re-run the bench; the score for the targeted case must rise and nothing else may regress. If a metric can't see the defect, the defect is also a gap in the bench — add a probe/metric (this is how the bench gets sharper).
6. **Record**: append a dated entry below.

## What the bench can and cannot see (read before trusting a score)

- The **reference renderer** (`benchmark/reference.py`) is vanilla headless Chromium with no stealth and a fixed selector list. It only captures a ~15-word shell from JS-hydrated / lightly-bot-protected gov CMS pages (e.g. ATO). So reference-based metrics (char_coverage, token_f1, noise — 70% of the composite weight) are **untrustworthy on JS-heavy pages**; supacrawl's own stealth scrape gets far more. Such cases are marked `scored: false` capability probes whose reliable signal is `expect_hit` + table structure (PIPES) + `word_spacing`. Hardening this renderer (hydration wait, or accepting supacrawl-as-reference for these) is an open thread.
- A large PDF scraped **concurrently** inside a bench run occasionally truncates to 0 words under resource contention with the reference-renderer browser; the same PDF scrapes perfectly standalone. Treat a single 0-word PDF result as suspect, not a regression, until reproduced standalone. Bench concurrency robustness is open.
- Reference-free metrics (`expect_hit`, `expect_absent`, `word_spacing`, structure counts) are trustworthy everywhere — lean on them for JS/gov/PDF cases.

## Lessons log

### 2026-06-15 — PDF word-spacing fix, word_spacing metric, real-world corpus (2026.6.1)

**Did.** Fixed a PDF inter-word-spacing defect (`services/pdf.py`: `extract_text()` default `x_tolerance=3` fuses words on LaTeX PDFs → set `x_tolerance=2`, `PDF_TEXT_X_TOLERANCE`). Added a reference-free `word_spacing` bench metric (catches exactly that defect; weight 0.10 in the composite; CJK/Arabic exempt). Added real-world corpus cases (frozen ATO gov PDF, two ATO HTML pages, Pet Circle, GitHub).

**Worked.** `x_tolerance=2` took the arXiv test PDF from ~5,300 fused tokens to ~9,400 correctly-spaced words and did not regress digitally-generated (gov) PDFs. The fix lifted the scored bench 72.2 → 86.8 in concert with the already-committed JS-shell (#126) and PDF-detection (#127) fixes (pdf 0→100, spa 48→83, infinite_scroll 14→88). Post-change baseline ~84.3 (the dip vs 86.8 was the inherently-flaky infinite-scroll case, not a regression).

**Surprised.** (1) The bench scored the _broken_ arXiv PDF 100/100 before — a blind spot, because for PDFs only `expect_hit` applied and the anchors still matched garbled text. The `word_spacing` metric closes that class of blind spot; look for others (a metric that can't see a defect is a bench gap). (2) The reference renderer can't see JS gov pages (above). (3) The CLI scrape envelope is `{success, data:{…}}` — content lives under `data`, not top level; a tooling helper that read top-level keys produced false "empty" results. (4) `time.monotonic()`-style sentinels and concurrency surprises recur (see project memory).

**Open threads (→ scoped as the `adaptive` module + the OOB-hardening issues).**

- No runtime quality signal exists; only `ScrapeResult.warnings` (`models.py:380`). Agents can't tell a shell/block/garble from a clean result. → Runtime Quality Assessment.
- Auto-escalation is one tier (`scrape.py:808-850`, Playwright→Patchright; an `if not self._stealth` guard blocks a 2nd hop to Camoufox). The working multi-tier pattern already exists for HTTP/2 errors (`scrape.py:1089`). Hard sites (legislation.gov.au 403, reddit/realestate 429) only succeed with manual `--engine camoufox`/`--wait-for`. → Adaptive auto-escalation.
- supacrawl is stateless per request — it re-derives the right strategy every time. → Per-domain strategy memory (the moat vs stateless hosted crawlers).
- Search is broken out of the box (`DEFAULT_PROVIDERS=("brave",)`; silent keyless DDG fallback hits `lite.duckduckgo.com` with no browser headers → 403/202; `registry.py:144-153`, `duckduckgo.py:50`). → Search OOB.
- Bugs: reddit default path crashes instead of returning a clean error (`browser.py:878` `_expand_iframes` unguarded / CLI `asyncio.run` lacks a top-level handler); Amazon soft-404 (HTTP 404 shell) reports `success=true` (`_assemble_result` `scrape.py:1591`; `_looks_like_bot_block` only gates 403/429/503); `only_main_content` over-prunes when a sparse selector matches (`converter.py:819-834`, `content_filter.py` 5-word floor). → folded into the Quality-Assessment and Auto-escalation issues.

### 2026-06-15 — Scoped the self-improving initiative (#135) + competitive validation

**Did.** Locked the flywheel (runtime quality signal → adaptive auto-escalation → per-domain strategy memory → experiential dev loop) and raised it as issues #128–135. Ran a competitive scan (Firecrawl, Crawl4AI, ScrapeGraphAI, Jina, Apify, Browserbase/Stagehand, Exa, Tavily).

**Validated.** No competitor ships cross-request per-domain strategy learning — their "state" is content caching, cookie/session persistence, or within-one-job adaptivity. Per-domain memory (#130) is a genuine moat for a local-first tool. Recommended design: a cost-aware champion per domain (the cheapest strategy whose EWMA quality stays above the bar), epsilon-greedy _downward_ exploration (probe cheaper strategies as sites drop defences), TTL/decay, local store under `.supacrawl/strategies/`. The #128 quality verdict is the reward label; the #129 ladder is the action space.

**Surprised / caveats.** (1) Crawl4AI's v0.7 blog markets per-domain persistence but its shipped API doesn't implement it — read their source before any public "nobody does this" claim. (2) Exa is ahead of us on signalling: it returns a per-URL error taxonomy (`SOURCE_NOT_AVAILABLE`, `CRAWL_TIMEOUT`), not just content → #128 should emit a verdict taxonomy, not a lone score. (3) Apify sets the first-hit defaults bar (auto HTTP↔browser + session rotation on CAPTCHA) → #129 must match it before memory helps. (4) MCP is now table stakes across all competitors, not a differentiator.

**Process note.** A web-research sub-agent that recursively fanned into 4 streams ran ~13 min with no intermediate output and read as a stall. Scope research agents tighter, or run them in the background with a heartbeat — don't fire opaque long-runners. The follow-up gap-closing used three tightly-scoped, no-fan-out agents that each returned in ~60-100s.

**Gap-closing follow-up (same day).** (a) Runtime quality signals (→ #128): build as gate-then-grade — a verdict taxonomy (BLOCKED_CHALLENGE/CAPTCHA/PAYWALL/JS_SHELL/THIN/OK…) resolved first via cheap header/title/cookie regex (FlareSolverr-style Cloudflare/Akamai/DataDome/PerimeterX fingerprints), then a 0-100 score; add `trafilatura` (F1≈0.909) for extraction. (b) Per-domain memory (→ #130): design validated against the satisficing-bandit literature — keep epsilon-greedy-downward, add an n_min≈3 champion floor (cold-start) and treat a hard block as an instant champion crash (non-stationarity). (c) Crawl4AI: read their shipped source (v0.8.9) — the marketed per-domain learning is NOT in the code; phrase any public claim as "marketed, not shipped," not "unmatched in the field."

### 2026-06-20 — Built the self-improving flywheel (#128–136)

**Did.** Shipped the whole initiative as five logical commits. **#128** — a shared, gate-then-grade quality assessor (`supacrawl/quality.py`) with a verdict taxonomy + 0-100 score; the reference-free metric primitives moved out of `benchmark/metrics.py` into the shared module so the bench and the live signal share one definition; `success` is now honest (soft-404 / block / garbled-PDF / empty → `success=false`); hard-fail results are no longer cached. **#129** — replaced four scattered single-tier retries (bot-block, platform, expect, HTTP/2) with one quality-gated escalation driver (playwright→patchright→camoufox→camoufox+HTTP/1.1, bounded budget, keep-best), plus an only_main_content thin-content fallback, a guarded iframe expansion (the Reddit crash), and a CLI top-level handler. **#130** — `StrategyStore`, a cost-aware champion bandit under `~/.supacrawl/strategies/` (EWMA, cheaper-equal demotion / clearly-better upgrade, epsilon-greedy-downward, hard-block crash, TTL decay); seeds the ladder on a known domain, records each rung's outcome; on by default for CLI+MCP, `supacrawl strategy list|show|forget|clear`. **#132/#136** — keyless search now fails loudly with an actionable error (was a silent empty), DDG gets the browser header profile, Brave quota surfaced in `supacrawl_health` with failover. **#134** — bench reference renderer settles hydration before capture; large PDFs get their own concurrency lane + a 0-word isolated retry. **#131** — this loop + the `adaptive` product module.

**Worked.** The flywheel composes cleanly: the #128 verdict is the reward, the #129 ladder the action space, the #130 store the policy — each keys off the next without special-casing. Sharing the reference-free metrics between runtime and bench (one module, two consumers) made "the runtime verdict agrees with the bench score" a unit test, not a hope. Full non-e2e suite green at 1056 (from 985 baseline + ~70 new tests), ruff+mypy clean project-wide.

**Surprised.** (1) The "baseline" was NOT green: 8 search/locale tests failed locally because a new direnv `source_up` cascade leaked the operator's real `SERPER_API_KEY` / `SUPACRAWL_LOCALE` / `SUPACRAWL_SEARCH_PROVIDERS` into the test process. The asserted behaviours were right; the env was dirty → fixed with a hermetic autouse fixture (CI was green all along; only local was poisoned). (2) The headline #132 bug survived the first delegated pass: wiring DDG headers + failover did NOT fix the keyless silent-empty — DDG returns a **200 with no result-links**, so the fix had to be at the service level (empty + no API-keyed provider available → loud failure). Verify the actual end-to-end behaviour, not just that the obvious sub-fixes landed. (3) The gov-ato HTML cases still cannot be scored: a hydration-settle wait does not help because the ATO CMS serves headless Chromium an **Akamai bot-shell** regardless of settle time — the reference renderer needs anti-bot escalation, which is a separate change. The conservative call (keep `scored:false`, document the path) was right.

**Open threads.**

- **Live real-world validation.** The flywheel is unit-tested but the operator's real matrix (Amazon, Qantas, hotels, ATO gov, SERP, pet food) needs a live run to confirm the escalation ladder + memory actually recover those sites on defaults, and to seed a first bench baseline with the new honest-success in effect. This is exactly what the `improve-supacrawl` loop's step 3 is for.
- **Bench gov re-scoring** depends on giving `ReferenceRenderer` an anti-bot tier (Patchright/Camoufox) so it can reach Akamai-gated content; then `gov-ato-*` HTML can move to scored. Tracked in `benchmark/corpus/default.yaml`.
- **Memory beyond single scrapes.** Strategy memory is wired for the single-scrape entry points (CLI scrape, MCP scrape). Crawl and search-scrape construct `ScrapeService` internally without a store; threading it through would let a crawl learn a site's strategy on page 1 and seed the rest — a clear future lift.
- **`trafilatura` for extraction** (F1≈0.909) was deliberately NOT added (new heavy dep, NixOS friction); revisit only if a bench case shows the converter is the bottleneck, not the fetch.

### 2026-06-20 — Live-validated the flywheel, made the bench headline honest (new baseline 85.0)

**Did.** Closed the "live real-world validation" open thread above. Seeded a fresh bench baseline at HEAD with honest-success in effect, then probed the operator's real matrix live on defaults (no manual flags). Then fixed two bench blind spots the baseline exposed.

**Worked — the flywheel is real in the field.** 5/6 reachable real sites recovered on defaults: Qantas (ok, 791w), Booking (ok, escalated once), ATO WFH (ok 94, escalated from scratch), PetCircle (ok, first try). The one non-recovery was a dead Amazon ASIN honestly reported as `error_status` 404 — not a bot defeat. **Per-domain memory seeding confirmed live:** ATO pass 1 climbed the ladder (2 attempts, escalated, 10.8s); pass 2 started from the learned strategy (1 attempt, no escalation, 8.9s, same score 94). Search returns real results via serper. A genuine bot wall (web-scraping.dev/antibot/easy) escalates the full ladder then fails honestly (`bot_challenge`, 403) rather than faking success.

**Surprised — the headline was lying low, not high.** The raw baseline read 78.3, down 13 from the prior 91.4 run, but `bench compare` reported "no significant changes." The drop was two artefacts, not a regression: (1) **degenerate reference renders.** `ecommerce-books-static` scored 57.6 despite a perfect scrape (266 words, expect_hit 1.0, char_coverage 1.0) because the reference renderer captured only **11 words** — so token_f1 (0.08) and noise (0.96), 40% of the composite, punished correct content the reference missed. Fixed with `reference_is_degenerate()` (a tiny reference paired with a much larger scrape): the runner discards the reference-based metrics and the case scores on the trustworthy reference-free signals (coverage/anchors/structure/spacing), lifting it to 91.7. Genuinely-tiny pages (`static-example`, ref==md==19, f1=1.0) and real under-extraction (`webscrapingdev-product`, healthy 341-word ref) are left correctly scored. (2) **an unbeatable wall in the scored index.** `antibot-webscrapingdev-easy` returns 403 to the full ladder incl. camoufox, while `web-scraping.dev/products` scrapes cleanly on defaults — a genuine evasion ceiling, not throttle. Reclassified `stable: false` (capability probe), same as g2. **New honest baseline: 85.0** (run `20260620T...-66a7b79`, 17 scored). The remaining real weak cases are `infinite-quotes-scroll` (14.9, got 5 words — scroll not triggering; flaky/real, a genuine target) and `webscrapingdev-product` (62, real under-extraction, ref=341>md=85).

**Open threads.**

- **infinite-scroll is a real target now**, not just "flaky": 5 words extracted vs 264 reference, expect_hit 0.00. The scroll-to-load path is not firing on `quotes.toscrape.com/scroll`. Investigate whether the escalation ladder's longer-wait rungs trigger scroll, or whether an explicit scroll action is needed.
- **`webscrapingdev-product` under-extraction** (85/341 words) — `only_main_content` may be over-pruning this product layout despite the thin-content fallback. Worth a converter look.
- **Bench gov re-scoring** still depends on an anti-bot tier in `ReferenceRenderer` (unchanged; the degeneracy guard now also auto-protects these — gov-ato-tax-rates/wfh have ref words 38/46 — though they remain explicitly `scored: false`).
- The reference renderer's intermittent under-capture (books-static got 11 words this run; it may render fully on another run) means the degeneracy guard is a robustness net, not a substitute for a stronger renderer — the durable fix is still hardening `reference.py`.
